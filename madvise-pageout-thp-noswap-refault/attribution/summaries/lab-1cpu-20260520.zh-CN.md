## 最重要结论

lab 结果复现了本地发现的关键 caveat：

| mode | kernel | `AnonHugePages` before `MADV_PAGEOUT` | `THPeligible` | `split_folio_to_list` |
| --- | --- | ---: | ---: | ---: |
| default | `v6.12.77` | 0 kB / 0 kB | 1 / 1 | 0 |
| default | `v6.19.9` | 16384 kB / 16384 kB | 1 / 1 | 16 |
| hugepage | `v6.12.77` | 0 kB / 0 kB | 1 / 1 | 0 |
| hugepage | `v6.19.9` | 16384 kB / 16384 kB | 1 / 1 | 16 |
| nohugepage | `v6.12.77` | 0 kB / 0 kB | 0 / 0 | 0 |
| nohugepage | `v6.19.9` | 0 kB / 0 kB | 0 / 0 | 0 |

因此，原始 formal lab timing 中的 old-faster 信号很可能不是“两个版本处于相同
actual THP backing 状态时 v6.19 更慢”，而是和 `v6.19.9` 实际获得 THP backing、
进入 no-swap reclaim split/failure path 有关。

## 解释

这轮 lab attribution 支持一个更窄、更谨慎的结论：

```text
default/hugepage 请求下，v6.19.9 实际 THP-backed，并在 no-swap MADV_PAGEOUT
reclaim 中反复进入 split_folio_to_list()；v6.12.77 没有实际 THP backing，也不
进入 split_folio_to_list()。在 nohugepage 同状态对照中，old-faster 信号消失。
```

因此，原先上游报告中的“THP/no-swap refault workflow regression”需要修正。当前
不宜继续把它表述为一个已经证明的 same-state 性能回归。更合适的 follow-up 是：

- 承认无 swap 场景不是真实 pageout/refault。
- 说明 `cycle_ns_per_page` 是 wall-clock iteration ns/page。
- 报告 lab ftrace/smaps 发现的 actual THP backing 差异。
- 说明在 `nohugepage` same-state control 中没有 old-faster。
- 询问 maintainer：是否仍值得从“no-swap THP split path 可否 fast-fail/skip split”
  角度继续做一个优化/行为讨论，而不是继续按原 regression claim 推进。
